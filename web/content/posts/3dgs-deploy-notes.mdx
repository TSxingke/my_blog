## 先聊两句：3DGS 在干什么？

如果用一句话概括 **3D Gaussian Splatting（3DGS）**：它在三维空间里摆了一堆**可学习的高斯椭球**（每个球有位置、协方差、不透明度、视角相关的颜色等），渲染时用**可微的 splatting**把这些「软乎乎的小雪球」投影到屏幕上，再用梯度下降把场景拟合出来。

我更喜欢用**抛雪球**来直觉化理解（严肃读者请把下面当成比喻，不是数学定义）：

1. **捏雪球**：初始化/优化阶段，你在空间里「捏」出一堆高斯——它们该在哪、该扁还是圆、该多不透明，都在学。
2. **抛雪球**：渲染阶段，把这些高斯往相机平面上「抛」过去——溅开、叠在一起，形成你看到的图像。

想快速建立画面感，推荐这支 B 站视频（讲解节奏友好）：[3D Gaussian Splatting 相关视频](https://www.bilibili.com/video/BV1zi421v7Dr/?spm_id_from=333.337.search-card.all.click)。

## 我动过的两处代码

### 1. 视频转图像：自己写 `extract_frames.py`

仓库自带的视频转图像流程在我这边会触发**不理想的裁切/抽帧行为**（具体症状略去五百字，总之结果不对）。我单独写了一个 `extract_frames.py`，用 OpenCV 按帧导出 JPEG，并支持采样步长、质量等参数，避免「神秘默认行为」。

核心思路很简单：打开 `VideoCapture`，按需要隔帧保存，`cv2.imwrite` 时带上 JPEG 质量与渐进式选项——**可控性**比「一行 ffmpeg 魔法」更适合排错。

若你使用同一脚本，请根据项目布局把输出目录指到 COLMAP / 3DGS 期望的 `input` 结构下。

### 2. 显存：`scene/cameras.py` 不把 GT 图常驻 GPU

默认实现里，真值图像会 `.to(self.data_device)`，训练时长期占用显存。若你数据集分辨率不低、批量或并行再上来一点，显存会肉眼可见地吃紧。

我把 GT 留在 CPU（默认 tensor 所在设备），需要时再搬到 GPU。对应修改如下（仅展示关键行）：

```56:57:/home/nsk/Project_python/gaussian-splatting/scene/cameras.py
        #self.original_image = gt_image.clamp(0.0, 1.0).to(self.data_device)
        self.original_image = gt_image.clamp(0.0, 1.0)
```

**注意**：这不是「白捡显存」——GT 仍在 **PyTorch 张量**里，只是默认留在 **CPU**，因此**系统内存（RAM）占用会上去**；换来的是 **GPU 显存（VRAM）压力下降**。训练时若仍要把图像搬到 GPU 算损失，还会引入 **CPU↔GPU 拷贝**与带宽开销，有时反而拖慢迭代。我这边是显存先顶不住，所以先这么改；若你显存宽裕或内存吃紧，请自行权衡，必要时再改回去。

## 跑通一条命令流水线

以下路径请替换成你自己的数据目录。

**1. COLMAP 稀疏重建 + 稠密（耗时大户）**

```bash
python convert.py -s data/bike_DJ_2/
```

这一步会跑 COLMAP 相关流程，**时间可能很长**。个人样本：约 **600 张 1080p 图像，耗时约 1 小时**（机器负载不同会差很多，仅供参考）。

**2. 训练 3DGS**

```bash
python train.py -s data/bike -m data/bike/output
```

`-s` 指向场景数据，`-m` 指向输出目录。训练过程中建议同时盯着**显存占用**与**loss 曲线**，别只看 tensorboard 漂亮。

## 数据采集：别只盯着分辨率

流程跑通之后，如果你打算认真拍数据而不是玩现成的 benchmark，可以记住下面几条**个人向**经验——**排序大概是**：

1. **视角覆盖与基线**：能不能把物体/场景从足够多的方向看清楚，比「单张 4K」重要得多。
2. **运动模糊与曝光**：糊成一团的高分辨率，只是更清晰地糊。
3. **分辨率**：在覆盖与清晰度过关的前提下，再往上加。

另外，原版 3DGS **在室外**往往比**室内**更「省心」——我个人的理解是：室内材质多、反射/间接光更复杂，而室外自然光下问题更「温和」。这不是定律，只是我踩过的统计印象。

## 小结

- **原理直觉**：把 3DGS 想成「捏雪球、抛雪球」有助于建立第一印象，真要写进论文还请回到高斯参数与可微渲染的严格定义。

- **工程**：抽帧用可控脚本；显存紧张时考虑 GT 图不默认上 GPU，但要接受 **RAM 上升**与可能的 **PCIe 拷贝**代价。
- **数据**：优先视角与光照条件，其次再卷分辨率（详见上文「数据采集」一节）。
